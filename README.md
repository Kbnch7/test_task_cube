# Task Management Service
Внутренний сервис для управления задачами команды разработки. Позволяет создавать задачи, отслеживать их текущий статус и управлять процессом выполнения.

# Технологический стек
Язык: Python 3.11

Фреймворк: FastAPI

База данных: PostgreSQL 14

ORM: SQLAlchemy 2.0 (асинхронный режим)

Кэширование: Redis 7.2

Контейнеризация: Docker / Docker Compose

Линтер: Ruff

Тестирование: Pytest + Httpx

# Как запустить
1. Подготовка окружения

Создайте файл .env в корневой директории и заполните его необходимыми переменными (примеры можно взять из параметров docker-compose.yml):

```.env
POSTGRES_USER=user
POSTGRES_PASSWORD=password
DATABASE_NAME=task_db
DATABASE_HOST=db
DATABASE_PORT=5432
DATABASE_URL=postgresql+asyncpg://user:password@db:5432/task_db
REDIS_URL=redis://redis:6379/0
```
2. Запуск через Docker Compose

Сервис поднимается одной командой:

Bash
docker compose up --build
После запуска API будет доступно по адресу: http://localhost:8000/api/v1/tasks/.
Интерактивная документация Swagger: http://localhost:8000/docs.

3. Запуск тестов

Для запуска тестов внутри контейнера или локально используйте:

```bash
pip install -r requirements.txt
pytest
```
# Принятые решения и архитектура
## 0. Layered Architecture

Решил использовать Layered Architecture для разделения кода на модули и слои. Позволяет легко подключать новый функционал или удалять старый, а также улучшает читаемость и поддерживаемость кода.

## 1. Структура данных (Модель Task)

Помимо базовых полей (название, описание, исполнитель), в модель добавлены:
id: UUID — использование UUID вместо простых ID повышает безопасность и облегчает миграции.
created_at / updated_at: Позволяют отслеживать время постановки задачи и её последнего изменения.

## 2. Кэширование в Redis

Реализовано кэширование двух типов:

Список задач: TTL установлен на 60 секунд (CACHE_TTL_TASKS). Это позволяет снизить нагрузку на БД при частом просмотре доски задач, сохраняя актуальность данных.

Отдельная задача: TTL составляет 300 секунд (5 минут). Данные конкретной задачи меняются реже, поэтому срок хранения выше.

Инвалидация: При обновлении (PATCH) или удалении (DELETE) задачи кэш для этой записи автоматически очищается, чтобы пользователь всегда получал актуальную информацию.

## 3. Валидация переходов статусов

В сервисе реализована проверка: если задача находится в статусе done, её перевод в любой другой статус невозможен. При попытке такого изменения API возвращает ошибку 400 Bad Request с пояснением "Task already done".

## 4. CI/CD Pipeline

В проекте настроен GitHub Actions (lint.yml), который автоматически запускает линтер Ruff при каждом пуше в ветку main. Это гарантирует соблюдение стандартов кодирования.

## 5. Healthchecks

Добавлены эндпоинты для проверки состояния сервиса (/health/readiness и /health/liveness), которые проверяют доступность базы данных PostgreSQL.

# Основные эндпоинты API
POST /api/v1/tasks/ — создание новой задачи.

GET /api/v1/tasks/ — получение списка всех задач (поддерживает пагинацию через skip и limit).

GET /api/v1/tasks/{id} — получение детальной информации о задаче по UUID.

PATCH /api/v1/tasks/{id} — обновление статуса задачи.

DELETE /api/v1/tasks/{id} — удаление задачи.

GET /health/readiness — проверка доступности базы данных PostgreSQL и Redis.

GET /health/liveness — проверка доступности сервера.
